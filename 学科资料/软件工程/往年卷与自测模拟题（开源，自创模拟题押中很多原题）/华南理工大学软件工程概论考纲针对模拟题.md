# 软件工程核心考点精选100题（含答案）
本文整理了软件工程学科核心考点，涵盖**判断题50道**与**选择题50道**，按知识模块分类排版，便于自测与复习。

## 一、判断题（50道）
### 第一组：软件基础与生命周期 (L1-L2)
1. 软件危机是因为软件发展滞后于硬件而产生。 ( )
2. 软件工程的本质是管理方法 + Best 技术 + 工具。 ( )
3. McCall 质量模型分为产品转移、运行和修订三个维度。 ( )
4. CMMI 1 级被称为“初始级（Initial）”。 ( )
5. 问题定义是软件生命周期“定义”阶段的第一步。 ( )
6. SRS 是指软件需求规格说明书。 ( )
7. V 模型的特点是测试与开发同步进行。 ( )
8. 原型模型最适合需求不明确的项目。 ( )
9. 螺旋模型的主要特点是强调“风险分析”。 ( )
10. 喷泉模型主要针对面向对象开发，具有无间隙性。 ( )

---

### 第二组：需求分析与 UML (L3)
1. **DFD（数据流图）**中，外部实体用矩形框表示。 ( )
2. ER 图中的菱形代表属性。 ( )
3. UML 4+1 视图中，“场景”视图是为了满足用户的需求。 ( )
4. 用例图中，extends 代表基础用例必须包含扩展用例。 ( )
5. 类图中，实线箭头表示继承关系（Ancestry）。 ( )
6. **聚合（Aggregation）**关系中，整体和部分可以不同时存在。 ( )
7. **组合（Composition）**关系比聚合关系耦合度更低。 ( )
8. **VCG（环路复杂度）**可以用来衡量程序的路径复杂程度。 ( )
9. 状态图中的 entry 代表进入状态时做的动作。 ( )
10. 活动图中，菱形通常代表决策或分支。 ( )

---

### 第三组：系统设计与原则 (U5, L8)
1. 系统设计分为概要设计和详细设计。 ( )
2. 详细设计的内容包括 DB 设计书、界面设计、伪码等。 ( )
3. 高内聚是指一个模块内部执行多个互不相关的任务。 ( )
4. 低耦合是指模块之间的联系应尽可能紧密。 ( )
5. **OCP（开闭原则）**是指对扩展开放，对修改关闭。 ( )
6. **DIP（依赖倒置原则）**要求针对接口编程，而不是针对实现编程。 ( )
7. **LOD（迪米特法则）**也叫最少知识原则，主张模块应“羞涩”。 ( )
8. **SRP（单一职责原则）**规定一个类只负责一项职责。 ( )
9. 状态模型中，after(5sec) 是一种基于时间的自动触发事件。 ( )
10. 软件结构中，模块的“深度”越大越好。 ( )

---

### 第四组：软件测试 (L6)
1. Pareto 原则认为 80% 的错误集中在 20% 的程序中。 ( )
2. 黑盒测试不需要看源代码。 ( )
3. 白盒测试包含路径覆盖、条件覆盖、逻辑覆盖。 ( )
4. 单元测试主要是针对 Core 核心代码进行的测试。 ( )
5. 集成测试的“三明治”策略结合了自顶向下和自底向上。 ( )
6. 系统测试是在集成测试确认后进行的。 ( )
7. 验收测试中，$\alpha$ 测试通常在开发者环境下进行。 ( )
8. $\beta$ 测试由用户在实际使用环境下进行。 ( )
9. 回归测试是为了确保修改旧错误时没有引入新错误。 ( )
10. VCG 计算时，公式一般为 $V(G) = 区域数 = 边 - 节点 + 2$。 ( )

---

### 第五组：维护与指标 (L7, L9, L11)
1. 改正性维护是为了修复测试中没发现的潜藏错误。 ( )
2. 适应性维护是为了让软件适应 OS 或数据库的升级。 ( )
3. 预防性维护是为了提高软件未来的可维护性。 ( )
4. 重构会改变代码的外部行为。 ( )
5. 再工程是对旧系统的全面翻新和文档重写。 ( )
6. LOC 是指代码行数（Lines of Code）。 ( )
7. FP 是指功能点（Function Points）。 ( )
8. 软件质量指标包括可靠性、性能、可维护性等。 ( )
9. 可靠性指标通常指在规定时间内程序不崩溃的概率。 ( )
10. 再工程属于正向工程，重构属于逆向工程。 ( )

## 模拟题1：公共设施报修管理系统（路面坑洼跟踪与修补）
### 一、核心考点：UML建模（用例图、活动图、类图）+ 用例描述
#### 1. UML用例图（核心要素）
- **参与者**：市民、系统管理员、维修人员
- **用例及关联**：
  - 市民 → 上传坑洼照片（仅市民可操作）
  - 系统管理员 → 管理坑洼信息、生成修补订单（仅管理员可操作）
  - 维修人员 → 提交修补资源信息（仅维修人员可操作）
- 用例图示意图（文字版）：
```
          ┌──────────┐        ┌──────────────┐        ┌──────────┐
          │  市民    │         │ 系统管理员   │        │ 维修人员  │
          └────┬─────┘        └──────┬───────┘        └────┬─────┘
               │                     │                     │
               ▼                     ▼                     ▼
        ┌────────────────┐  ┌────────────────┐  ┌────────────────┐
        │ 上传坑洼照片    │  │ 管理坑洼信息    │  │ 提交修补资源信息 │
        └────────────────┘  │ 生成修补订单    │  └────────────────┘
                            └────────────────┘
```

#### 2. 用例描述（以“生成修补订单”为例）
| 要素         | 内容                                                                 |
|--------------|----------------------------------------------------------------------|
| 用例名称     | 生成修补订单                                                         |
| 参与者       | 系统管理员                                                           |
| 前置条件     | 管理员已登录，系统中存在未分配的坑洼记录                             |
| 基本流       | 1. 管理员筛选特定区域的坑洼信息；<br>2. 系统显示符合条件的坑洼列表；<br>3. 管理员勾选多个坑洼，点击“打包订单”；<br>4. 系统生成唯一订单号，要求选择分配的维修小组；<br>5. 管理员确认分配，系统更新坑洼状态为“维修中” |

  | 类名          | 属性                          |
  |---------------|-------------------------------|

## 模拟题2：智能陪聊机器人“小伴”软件策划（AI童伴）
### 一、核心考点：项目策划+需求分析+概要设计+质量保障
#### 1. 项目基础信息
- 项目名称：“AI童伴” (AI-Buddy) —— 启发式儿童智能陪伴系统
- 项目定位：基于生成式AI的语音聊天软件，目标用户4-10岁儿童，核心价值：日常对话+价值观引导+语言练习+家长情绪监控。

#### 2. 需求分析（区分功能/非功能）
| 需求类型   | 具体内容                                                                 |

## 通用学习建议（重点强调）
1. **类图关系区分**：
   - 泛化（继承）：A是B的一种（如User是Account的一种）→ 实线+空心三角；
   - 组合/聚合：A是B的组成部分（如订单包含多个坑洼）→ 实线+实心/空心菱形；
   - 关联：A和B有业务联系（如User和Item）→ 普通实线。
2. **需求分析边界**：功能需求回答“系统能做什么”，非功能需求回答“系统要做到什么程度/满足什么约束”。
3. **QA计划核心**：需结合项目场景（如校园/社区、户外/室内）设计具体、可量化的测试/保障手段，避免泛泛而谈。
